1 MPCitH

1.1 Original idea + other (mostly ignored proposals).

  • t-privacy: t parties can't collude to learn to witness. I.e. how many parties can the prover reveal to the verifier.

Input: Witness w, Public key x, public function f and a protocol \(\Pi\) that is \(t\) -private.

  1. Choose \(w_i \leftarrow\{0,1\}^m\) so that \(\sum w_i=w\). Prover runs the protocol \(\Pi(f,x,w_1,\ldots,w_n)\). And commits to the "views of the parties".
  2. Verifier chooses \(T\subset \{1,\ldots,n\}\), \(#T=t\).
  3. Prover opens the views of parties \(t\in T\).
  4. Verifier checks that the opened views are consistent with their commits and that they output correct.

1.3 Other proposals

Same algorithm as original, but MPC r-robust: Constant round.

  • Wishlist: Better MPC protocols compared to BGW.
    • SPDZ
    • Beaver triples: Objective: Known \((x_i,y_i)\) find \((z_i)\) so that: $ ∑ zi=(∑ xi)(∑ yi) Given triples \((a,b,c)\) and a sharing \((a_i,b_i,c_i)\) then, choose \(\varepsilon\) at random and do:
      • Compute \(\alpha_i=x_i-a_i\) and \(\beta_i=y_i-b_i\)
      • Open \(\alpha\) and \(\beta\), i.e
        • \(\alpha=\sum \alpha_i\)
        • \(\beta=\sum \beta_i\)
      • Compute: \(z_i=c_i-\alpha\cdot b - \beta \cdot a_i + \alpha \cdot \beta\)

2 Better solutions: Using correlated randomness

  • Preprocessing: BBQ, Rainier
    • It is possible to verify the result of an S-box in one operation: Let \((s_i)\in\mathcal{F}_{2^8}\) and random \((r_i)\in\mathcal{F}_{2^8}\):
    • Use the Beaver triples technique to compute a sharing of \(s\cdot r\) and open it
    • Compute \(s^{-1} \cdot r^{-1}\)
      • If conditionals are a pain in MPC, so we choose an input where all the S-boxes have \(\neq 0\) input.
      • Almost negligible effect on Key size (costs about 2-4 bits of security).
    • The parties compute their share of \(s^{-1}\) by: \(s^{-1} \cdot r^{-1} \cdot r_i\)
  • Sacrificing: Banquet
    • No need to compute the S-box, the prover gives the answer and a proof that it is correct.
      • Instead of using a triple to calcutate the answer, sacrifices one to prove the provided answer was correct.
      • Look like the same but the properties of the zk proof are tricky.
      • No need to reduce key space! Since inversion is not done in a multiparty way.

4 Optimizations that are already known.

  • Commits done in Merkle tree: Reveal takes \(\log{n}\) communication.
  • Efficent generation of random triples.
    • Can be done from seed, the prover calculates correction bits and gives them to the last party.
    • So communication is limited to 1 field element/gate + 1 seed (global).
    • Same for any other randomness:
      • Consequence: Randomness needs to be generated from a secure DRBG, otherwise the prover could choose the challenge.
      • Switching between a fast generator and a secure one could give a little bit of speed.
        • No communication differenece, so who cares.
  • Batching/Compressions.
  • Inserting output if evaluating a series of mult gates.
  • Sampling on the fly. Mostly unexplored.
  • Rain: Bigger S-box allows to work in larger fields.

5 Unexplored options

  • NIMPC: Probably really bad.
  • Could the last party proof it's been honest?
    • Already some work on this.
    • Paper of Nof proving: If the sharing has some redundancy, then the one party has low chance of cheating. Problem: Currently it requires leaving 2 parties unopened. What we need: Some sort of magical homomorphic (1-time) MAC.
  • AES-specific sacrificing.
  • Multlivariate Schwartz-Zippel.
  • Multiple verifiers!.
    • Used in MPC for better performance, could be ported to MPCiTH for the non-interactive (our) case.
