1 MPCitH

1.1 MPCitH: Original idea.

Input  
  witness w
  Public key x,
  Public function f
  Protocol \(\Pi\) that is \(t\) -private.
  1. Choose \(w_i \stackrel{unif}{\leftarrow} \{0,1\}^m\) so that \(\sum w_i=w\).
  2. Prover runs the protocol \(\Pi(f,x,w_1,\ldots,w_n)\).
  3. And commits to the "views of the parties".
  4. Verifier chooses \(T\subset \{1,\ldots,n\}\), \(\vert T\vert =t\).
  5. Prover opens the views of parties \(t\in T\).
  6. Verifier checks that the opened views are consistent with their commits and that they output correct.

1.2 Ligero's choice

There is an extra party I which can't be opened, but has certain special robustness properties.

Input  
  Witness w
  Public key x,
  Public function f
  Protocol \(\Pi\) that is \(t\) -private.
  1. Set \(w_i \leftarrow\{0,1\}^m\) so that \(\sum w_i=w\).
  2. Prover runs the beginning of the protocol \(\Pi(f,x,w_1,\ldots,w_n)\).
  3. And commits to the "views of the parties".
  4. Prover and verifier invoke a coin-flipping oracle and get a random challenge r.
  5. Prover finishes to run the protocol, commits to the views. And opens a subset challenged by the verifier.
  6. Verifier verifies.
  • Gain?: No idea. Assymptotically better somehow.
  • Assymptotics not good enough for signature. Has constant overhead!. (Communications= 0(1)+extra(\(\kappa\)).

1.3 Better solutions: Using correlated randomness.

  • Goal: Use correlated randomness.
  • Wishlist: Better MPC protocols compared to BGW.
    • SPDZ
    • Beaver triples: Objective: Known \((x_i,y_i)\) find \((z_i)\) so that: \(\sum z_i=(\sum x_i)(\sum y_i)\), without sharing \(x\) or \(y\). Given triples \((a,b,c)\) and a sharing \((a_i,b_i,c_i)\) then, choose \(\varepsilon\) at random and do:
      • Compute \(\alpha_i=x_i-a_i\) and \(\beta_i=y_i-b_i\)
      • Open \(\alpha\) and \(\beta\), i.e \(\alpha=\sum \alpha_i\) \(\beta=\sum \beta_i\)
      • Compute: \(z_i=c_i-\alpha\cdot b - \beta \cdot a_i + \alpha \cdot \beta\)

1.4 KKW

Idea: Randomness is verified for some parties before the execution of \(\Pi_f\), and the verifier trusts that if the opened parties are honest, so are the unopened ones.

1. Preprocessing phase
2. Generate \(M\cdot (\text{number of multiplication gates})\) random triples \((x_i,y_i,z_i)_{i=0}^N\).
3. Verifier asks to see the randomness assigned to \(M-\tau\) parties. Verifies that the triples are correct.
4. Execute IKOS algorithm on \(\tau\) unopened parties.
  • Parameters:
    1. N: Number of parties.
    2. M: Number of preprocessing rounds.
    3. \(\tau\) cut parameter: Determines how many rounds of protocol will be executed in the end. To be chosen for optimal soundness.
  • An obvious optimization:
    1. Generate random triples from seed \(sd\), and correct \(z_n\) to correct value: The prover need only give the verifier \(sd_i\), which is the seed used by party \(i\). It can be generated by \(sd_i=H(sd||i)\).

1.6 KKW Part 2

  • Soundness: \[ \displaystyle{ \stackrel{min}{0\leq\tau\leq M}\;\; \stackrel{max}{0\leq c\leq M-\tau}\;\; \left\{ \frac{ \binom{M-c}{\tau} }{ \binom{M}{\tau} N^{M-\tau-c} } \right\} } \]
  • Communication complexity:

\[ 2\vert hash \vert + \tau \vert sd \vert + \vert hash \vert + (M-\tau)(\vert com \vert) + 3\log_2{\vert\mathbb{F}\vert} + \vert sd \vert + (M-\tau)(log N \vert sd \vert) + (M-\tau)(\vert com \vert + (M-\tau)\log_2{\vert\mathbb{F}\vert}(3n_{mult}+ 2n_{sq} + n_{in}+1) \]

  • Observation: \(\vert hash \vert\geq 2\cdot \kappa\), where \(\kappa\) is the security level.

1.7 BN

Idea: Randomness verified is during execution of the circuit instead of before.

  1. It is possible for the verifier to check that a multiplication has been done correctly by inserting some randomness.

    \(\sum z_i=(\sum x_i)(\sum y_i)\), without sharing \(x\) or \(y\).
    Given triples \((a,b,c)\) and a sharing \((a_i,b_i,c_i)\) then
    choose \(\varepsilon\) at random and do:
    1. Compute \(\alpha_i=\varepsilon x_i-a_i\) and \(\beta_i=y_i-b_i\)
    2. Open \(\alpha\) and \(\beta\), i.e
    \(\alpha=\sum \alpha_i\)
    \(\beta=\sum \beta_i\)
    3. Compute:
    \(v_i=\varepsilon z_i -c_i-\alpha\cdot b + \beta \cdot a_i - \alpha \cdot \beta\)
    4. Check that \(v=\sum v_i=0\).

1.8 BN Part 2

  1. BN algorithm:
    1. Imperfect preprocessing:
      1. Generate random triples. No inmediate challenge.
      2. Run the circuit and save the output value of each gate.
      3. Commit to the "state". That is: The seed used for each party, the correction bits and the output of each gate. Important: Prover sends one commit string overall. We call this commit \(h_\Gamma\).
    2. Challenge: The verifier chooses a random seed, used to generate the challenges.
    3. Prover computes challenges on all gates, commits to the result of each party and sends it to the verifier. We call this commit \(h_\pi\).
    4. Verifier chooses to all-but-one party per round.
    5. Prover sends the requested parties plus the complementary commits necessary to compute the hash tree.
    6. Verifier uses the data given by the verifier to compute the global hashes.
  2. Parameters:
    1. N: Number of parties.
    2. M: Number of preprocessing rounds.

1.9 BN Part 3

  1. Soundness:

\[ \left( \frac{2N+\vert\mathbb{F}\vert-2}{\vert\mathbb{F}\vert N} \right)^M \]

  • Communication complexity:

$$ | hash | + | hash | + | sd | + M(| sd | + | com | + 4log2(|\mathbb{F}|)nmult

  • 3log2(|\mathbb{F}|)nsq+log2(|\mathbb{F}|)nin + 2log2(|\mathbb{F}|)) $$

3.1 TODO Better solutions for AES.

  • Preprocessing: BBQ, Rainier

    • BBQ

      • It is possible to verify the result of an S-box in one operation: Let \((s_i)\in\mathbb{F}_{2^8}\) and random \((r_i)\in\mathbb{F}_{2^8}\):
      • Use the Beaver triples technique to compute a sharing of \(s\cdot r\) and open it
      • Compute \(s^{-1} \cdot r^{-1}\)
        • If conditionals are a pain in MPC, so we choose an input where all the S-boxes have \(\neq 0\) input.
        • Almost negligible effect on Key size (costs about 2-4 bits of security).
      • The parties compute their share of \(s^{-1}\) by:

      \(s^{-1} \cdot r^{-1} \cdot r_i\)

    • Rain: AES with larger S-box to reduce the number of rounds.

3.2 Sacrificing: Banquet

  • No need to compute the S-box, the prover gives the answer and a proof that it is correct.
  • Instead of using a triple to calcutate the answer, sacrifices one to prove the provided answer was correct.
    • Let \(m\) be the number of S-boxes.
    • Let \(s_1^{(i)},\ldots,s_m^{(i)}\) be the outup of the S-boxes and \(t_1^{(i)},\ldots,t_m^{(i)}\) be sharing of the inverses.
    • Goal: Verify that, \[ \left(\sum_j s_j^{(i)}\right) \cdot \left(\sum_j t_j^{(i)}\right) = 1,\;\; \forall j\in [m] \]

3.3 Sacrificing: Banquet. Part 2.

  • Let \(r_i\) be the challenges of the verifier.
  • Let \(m\) be the number of S-boxes, (i.e 10?/party).
    • Choose \(m_1,m_2\) so that: \(m=m_1\cdot m_2\).
    • Divide the vector \((s_1^{(i)},\ldots,s_m^{(i)})\) into \(m_2\) chunks of size \(m_1\). \[ s_k^{'(i)}=(r_1s_{1+(k-1)m_1},\ldots, r_js_{m_1+(k-1)m_1} ) \;\;\forall k\in [m_2] \]
    • Similar for \(t\). \[ t_k^{'(i)}=(t_{1+(k-1)m_1},\ldots, t_{m_1+(k-1)m_1} ) \;\;\forall k\in [m_2] \]
    • The point: \[ \left\langle s_k^{'(i)}, t_k^{'(i)} \right\rangle =\sum_{j\in [m_1]} r_j,\;\;\forall k \in [m_2] \]

3.4 Sacrificing: Banquet. Part 3.

  • Build polynomials, \(P_j\), \(S_j\) and \(T_j\) so that: \[ (S_{1}(k),\ldots,S_{m_2}(k))=s_k^{'(i)};\; (T_{1}(k),\ldots,T_{m_2}(k))=t_k^{'(i)};\; \] \(S_k(m_2)\;,T_k(m_2)\) to be chosen at random. \[ P=\sum_{j\in[m_1]} S_j T_j, \;\; \forall k\in [m_2] \] And \(S(l),T(l)\) are chosen at random. And \(P(l)=S(l)\cdot T(l)\).
  • Lift \(P,S,T\) to a field \(\mathbb{F}_{2^\lambda}\) with \(\lambda\) to be chosen large enough.
    • Choosing \(\lambda\) is not free as \(dim (\mathbb{F}_{2^\lambda}/\mathbb{F}_{2})=\lambda\). So $λ$-bits of information are needed per element in the field. (\(3\lambda\cdot (m+1)\) for the polies for example).
    • Choose if \(R\in\mathbb{F}_{2^\lambda}-[m_2]\) at random. Check \(P(R)=S(R)\cdot T(R)\).

3.5 Sacrificing: Banquet. Part 4. Some final remarks.

  • Observation: We can't choose \(R\in[m_2]\) because at those points the equality is guaranteed to hold by construction.
  • Observation: Interpolation is linear, hence the parties can compute a share of \(S_j(R)\) without comunication.

3.9 Optimizations that are already known.

  • Commits done in Merkle tree: Reveal takes \(\log{n}\) communication.
  • Efficent generation of random triples.
    • Can be done from seed, the prover calculates correction bits and gives them to the last party.
    • So communication is limited to 1 field element/gate + 1 seed (global).
    • Same for any other randomness:
      • Consequence: Randomness needs to be generated from a secure DRBG, otherwise the prover could choose the challenge.
      • Switching between a fast generator and a secure one could give a little bit of speed.
        • No communication differenece, so who cares.
  • Batching.
    • Take random linear combinations of elments, i.e amortized proofs. Soundness is reduced, but communication is much better.
  • Inserting output if evaluating a series of mult gates.
  • Sampling on the fly. Mostly unexplored.
  • Extendeded witness.
    • Prover adds elements to the witness that prove specific property. E.g: In BN: Extended witness contains a zk-proof that \(w\) is binary (i,e \(w=(w_1,\ldots,w_n),w_i\in\{0,1\}\)).
  • No need to transmit \(\beta\) when sacrificing!

3.10 Some delicate stuff.

  • Since only one plaintext/ciphertext pair is revealed, less rounds are necessary to guarantee \(128\) bits of security.
  • Rain: Bigger S-box allows to work in larger fields.
